---
title: "Manifest Targets Configuration"
description: Defining compile rules, dependency linkages, and default execution targets within mokai.toml.
---

Targets represent the physical build blueprints of your project graph. Each target specified inside a workspace or local dependency manifest tells `mokai` how to ingest structural compilation paths, link libraries, and surface executable outputs.

### Target Properties Reference

| Property | Type | Description |
| :--- | :--- | :--- |
| `name` | `string` | The distinct alphanumeric label used to identify the target output artifact. |
| `type` | `string` | The target compilation classification rule. Must evaluate to either `"executable"` or `"library"`. |
| `is_default` | `boolean` | *(Optional)* If set to `true`, marks this executable target as the primary fallback execution target for `mokai run`. |
| `src` | `array<string>` | Directory structures or relative source file matching rules parsed during graph mapping. |

### Designing the Execution Fallback Chain

When multiple binary targets exist inside a highly integrated complex layout, `mokai` follows a rigid deterministic lookup route to fulfill the execution context when no explicit name parameter is provided to the CLI runner:

```
[mokai run invoked without args]
               │
               ▼
┌──────────────────────────────┐
│  Looks for target where      │
│  is_default = true           │───────► [Target Found] ──► Compile & Launch
└──────────────┬───────────────┘
               │ (None explicitly flagged)
               ▼
┌──────────────────────────────┐
│  Selects the first declared  │
│  Executable target type      │───────► [Target Found] ──► Compile & Launch
└──────────────┬───────────────┘
               │ (No executables found)
               ▼
┌──────────────────────────────┐
│ Return unexpected CliError   │
│ (InvalidArguments Code)      │
└──────────────────────────────┘
```

### Manifest Example Configuration

The following blueprint snippet illustrates how a root project maps local links (like a local `{fmt}` framework module) alongside multiple execution rules:

```toml
[project]
name = "my_workspace"
version = "0.0.1"
cpp_version = "c++23"

dependencies = [
    "../fmt"
]

[[targets]]
name = "fmt_lib"
type = "library"
src = ["src/lib/**/*.cpp"]

[[targets]]
name = "testfmt"
type = "executable"
is_default = true
src = ["src/main.cpp"]

[[targets]]
name = "benchmarks"
type = "executable"
is_default = false
src = ["src/bench.cpp"]
```

### Validation Rules

* **Library Restrictions:** Mapped library targets (`type = "library"`) are structured purely for graph compilation routines and deep internal object linking. Passing a library target name into `mokai run` raises an explicit validation validation error block:
  ```text
  Target validation error: 'fmt_lib' is configured as a library rule module and cannot be run.
  ```
* **Unique Constraints:** Target names within the same localized project structure context must remain fully unique to block artifact overwrite paths inside `./build/`.
