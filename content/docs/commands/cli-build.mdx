---
title: "Mokai Command: build"
description: Execute multi-threaded structural graph compilation with zero-overhead dependency caching.
---

The `mokai build` command is the core execution driver of the Mokai build engine. It parses the localized dependency map, tracks file modification states across workspace boundaries, evaluates an optimal topological build order, and distributes compilation units across parallel worker threads.

### Usage

```bash
mokai build [workspace_path] [flags...]
```

* If `workspace_path` is omitted, Mokai defaults to the active current working directory (`.`).
* If a directory is specified, it scans that target for a valid `mokai.toml` manifest file before initializing the graph pipeline.

### Pipeline Pipeline Stages

```
   [Parse Arguments] ──► [getConfig Registry Match]
                                │
                                ▼
   [Spawn Compilation Worker Pool] ◄── [Topological Build Order]
                │
                ├──► Step [1/N]: Build Libraries (e.g., local Link ../fmt)
                └──► Step [2/N]: Build Executable Binaries (e.g., testfmt)
```

1. **Manifest Validation**: Resolves the root workspace manifest and recursively scans sub-dependency graphs through the absolute path registry map (`m_config_registry`), avoiding multiple reads for shared dependencies.
2. **Topological Sort**: Computes a dependency tree. If a cyclic or unresolvable rule layout is encountered, compilation falls out cleanly.
3. **Delta Evaluation**: Evaluates file hashes and timestamps to bypass already compiled `.o` / `.obj` translation units.
4. **Execution Parallelization**: Distributes remaining dirty compilation units across available hardware threads.

### Examples

**Standard Compilation (Debug Profile):**
```bash
mokai build
```

**Targeted Release Rebuild:**
Forcing an unconditional clean compilation run optimized for high-performance deployment:
```bash
mokai build --release --clean
```
