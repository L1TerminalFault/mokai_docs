Mokai eliminates implicit build-system magic. Every operational option, state flag, and command-line mutation acts as a deterministic functional mapper over raw hardware toolchain execution arrays. 

This reference provides exact breakdown mechanics for string injections, machine instruction generation hints, symbol generation schemes, and translation unit caching limits across different platforms.

### Global Driver Controls

The front-end state machinery evaluates arguments into a specialized structural definition known internally as `GlobalOptions`. Each command-line switch maps to explicit variable configurations that govern how background thread pools are spawned, how standard outputs are managed, and how translation target sets are sliced out of the dependency graph.

| Flag | Shorthand | Memory / Environment State Modification | Hard Driver Behavior |
| :--- | :--- | :--- | :--- |
| `--release` | *None* | Toggles structural build configuration to `BuildProfile::Release`. | Directs string formatting pipelines to abandon debug mapping matrices and switch to maximum machine-instruction packing. |
| `--debug` | *None* | Toggles structural build configuration to `BuildProfile::Debug` *(Default Engine Base)*. | Preserves un-optimized source tracking records and leaves translation sequences untouched for debugger hardware ingestion. |
| `--no-cache`<br/>`--clean` | *None* | Flushes localized timestamp registers and memory maps. Forces state tracker structures to evaluate all source dependencies as dirty context blocks. | Drops local tracking maps completely, bypasses existing target object validation checks, and re-triggers translation files back through the execution pool. |
| `--jobs <n>` | `-j <n>` | Sets a hard limit on background worker allocations inside the hardware compilation thread pool. | Explicitly bounds the count of active file descriptor channels and parallel sub-processes spawned via internal execution functions. |
| `--target <t>` | *None* | Slices down the processing pipeline array using string comparison checks. | Forces the topological sorting worker engine to skip target rule records whose name field fails to match the targeted sequence string. |
| `--verbose` | `-v` | Shifts the trace visibility controller matrix down to the internal debugging level (`log::Level::Debug`). | Disables trace choking and outputs the exact raw subprocess terminal command strings before execution pipelines run. |
| `--quiet` | `-q` | Shifts the trace visibility controller matrix up to the maximum error filtering boundary (`log::Level::Error`). | Suppresses standard telemetry metrics, build-time outputs, and timing logs. Only critical build-dropping terminal halts bypass this layout filter. |

### Code Profile Optimization & Code-Gen Layouts

When a profile boundary condition changes, Mokai alters how code optimization is requested, how standard library headers behave, and how error tracking loops are generated.

#### 1. Debug Configuration Pipeline

Debug builds prioritize linear instruction paths and rapid file writing over runtime execution speeds. The engine structures flags so that generated assembly instructions match the source code file locations $1:1$, keeping execution paths clear for low-level step-through analysis and line tracking.

##### GCC / Clang Toolchains
```bash
-g3 -O0 -fno-inline -D_DEBUG
```
* **`-g3`**: Generates exhaustive DWARF debug symbols. It includes macro expansion definitions within the object files, ensuring that complex code tracking tools can read embedded definitions natively.
* **`-O0`**: Completely cuts off optimization passes. It stops the compiler backend from reorganizing code structures, rearranging basic assembly block sequences, or removing variables from storage registers.
* **`-fno-inline`**: Blocks function expansion behaviors. It forces every functional unit call to retain its actual call layout, branch operations, and frame pointer structures inside the target assembly tree.
* **`-D_DEBUG`**: Defines the active runtime validation flag used by standard library components to turn on bounds checking inside core collection layouts.

##### MSVC Native Toolchain
```cmd
/Zi /Od /RTC1 /MDd /D_DEBUG
```
* **`/Zi`**: Writes full debug definitions out into a discrete Program Database file (`.pdb`). This scheme isolates debugging symbol overhead completely away from the actual translated binary image footprint.
* **`/Od`**: Stops compiler optimization threads from packing loops, removing unreferenced local parameters, or optimizing code registers.
* **`/RTC1`**: Injects hardware validation tracking functions to detect stack frame corruption, overruns, and uninitialized data references.
* **`/MDd`**: Directs target files to link explicitly against the debug-safe, multi-threaded dynamic C-Runtime allocation frameworks.

---

#### 2. Release Configuration Pipeline

Release builds configure target backends for absolute performance. Mokai passes aggressive inline packing instructions, turns off type introspection markers, and cuts away language exception tracking mechanisms to strip execution overhead to the absolute minimum.

##### GCC / Clang Toolchains
```bash
-O3 -march=native -ffast-math -fno-rtti -fno-exceptions -flto -DNDEBUG
```
* **`-O3`**: Turns on aggressive optimizations. The compiler backend uses parallel vectorization registers, unrolls loops, reorders basic machine instructions, and tracks global data behaviors to maximize throughput.
* **`-march=native`**: Queries the host system CPU layout at compile time to unlock architecture-specific vector processing extensions, like AVX2, AVX-512, or specialized cryptography instructions.
* **`-ffast-math`**: Relaxes strict IEEE 754 floating-point standards. This lets the hardware optimize mathematical operations via algebraic approximations and combined multiply-add instructions.
* **`-fno-rtti`**: Turns off the generation of Run-Time Type Information symbols. This completely cleans out class descriptor metadata structures, shrinking the binary file footprint.
* **`-fno-exceptions`**: Cuts away language try-catch execution frameworks. This removes the stack unwind tracking tracking table structures entirely, creating tighter assembly execution paths.
* **`-flto`**: Activates Link-Time Optimization routines. This defers the code generation step to the final linkage phase, letting the system optimize function calls across entirely separate source code modules.
* **`-DNDEBUG`**: Preprocessor flag that strips asset validation checks and macro debug traps out of source files before translation passes begin.

##### MSVC Native Toolchain
```cmd
/O2 /Oi /Gy /GL /Gw /MD /DNDEBUG
```
* **`/O2`**: Configures the platform backend to prioritize processing speeds over the size of the written binary output.
* **`/Oi`**: Forces the compiler to exchange standard math and system function calls for direct, high-performance assembly instructions.
* **`/Gy`**: Packages distinct functions as individual, packaged COMDAT record allocations. This gives the downstream linker the ability to dead-strip unreferenced routines.
* **`/GL`**: Turns on Whole Program Optimization behaviors. This enables deep cross-module performance profiling and cross-boundary analysis during final assembly link stages.
* **`/Gw`**: Optimizes and groups structural global variable definitions inside object files to let downstream optimization steps strip unreferenced global allocations.
* **`/MD`**: Configures the target output file to link cleanly against standard production multi-threaded dynamic C-Runtime system modules.

### Overriding Configuration Parameters via Manifest Files

Mokai keeps its automated optimization rules separate from custom build targets. When a compilation plan needs explicit instructions, platform adjustments, or specialized language components, you can pass parameters into the engine directly using the target array blocks inside your `mokai.toml`.

```toml
[project]
name = "low_level_core"
version = "1.0.0"
cpp_version = "c++26"

# Absolute injection block passed straight down to child compilers
user_compiler_flags = [
    "-fno-omit-frame-pointer",  # Keeps frame pointer register layouts trace-safe
    "-fno-plt",                 # Stops PLT call generation for direct function addressing
    "-msse4.2"                  # Enforces SIMD hardware stream instruction sets
]
```

### Append Evaluation Sequencing Order

To maintain predictable compilation configurations, Mokai applies flags using a strict append order. This design guarantees that your explicit `user_compiler_flags` take absolute priority and override any standard profile settings:

```
┌───────────────────────────┐
│  Mokai Language Spec Flag  │  (e.g., -std=c++23)
└─────────────┬─────────────┘
              │
              ▼
┌───────────────────────────┐
│   Profile Setting Flags   │  (e.g., --release -> -O3 -flto)
└─────────────┬─────────────┘
              │
              ▼
┌───────────────────────────┐
│    User Configuration     │  (Appended last: overrides or extends
│    Manifest Flag Array    │   previous compilation behaviors)
└───────────────────────────┘
```

This strict layout placement means that if you configure `-O0` within your manifest's `user_compiler_flags` array, it overrides the automatic `-O3` profile optimization flags safely without breaking language specification targets.
