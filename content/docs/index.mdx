Welcome to Mokai—a native, low-overhead build instrument, absolute dependency resolver, and static toolkit engine for modern C++. 

Mokai eliminates heavy multi-stage generator passes, replacing complex scripting frameworks with a single, highly structured structural layout context (`mokai.toml`). The core engine handles parallel graph compilation, local asset mapping pipelines, and absolute hardware execution loops without generating interstitial build cache bloat.

---

### Host Installation Profiles

Mokai is currently distributed via shell bootstrapping vectors. The release image targets POSIX subsystem capabilities, low-level process spawning engines (`posix_spawn`), and zero-cost virtual memory maps natively.

#### Installation via Curl Pipeline
To fetch the compiled engine toolchain binary directly from the centralized delivery matrix and inject it into your local environment path, execute the standard shell route:

```bash
curl -fsSL https://raw.githubusercontent.com/mokai-build/mokai/main/get.sh | sh
```

The script automatically detects your active hardware layout parameters, provisions the native executable target, and registers `mokai` inside your shell variable paths.

#### Verification and Telemetry Trace
Confirm the toolchain registration and verify binary ingestion by querying the structural driver engine directly:
```bash
mokai --version
```

### Initializing Your First Workspace Matrix

Bootstrapping a codebase layout requires zero mechanical configuration or boilerplate code tracking strings. 

#### 1. Execute the Scaffold Routine
Invoke the interactive template engine straight out of your target deployment directory context:
```bash
mokai create blueprint_app
```

This execution thread extracts hard-coded static skeleton snapshots directly out of embedded binary storage fields. It fires up raw terminal selection loops allowing you to pin language specifications up to the experimental edge (`c++26`), assign target generation blueprints, and map an empty, clean local Git version control tree (`git init --initial-branch=main`).

#### 2. The Blueprint Filesystem Shape
The initializer finishes instantly, structuring a clear workspace without dropping complex hidden meta-directory files:

```text
blueprint_app/
├── .git/               # Core Local Version Control Tree Index
├── mokai.toml          # Central Manifest Workspace Config Blueprint
└── src/
    └── main.cpp        # Zero-Boilerplate Standard Application Entry
```

### Manifest Ingestion Syntax (`mokai.toml`)

Mokai trades implicit logic scripts for a clear, standardized data format. Below is the blueprint syntax layout generated by the engine. It maps targets, sets the strict engine version parameters, and wires up external local directories without writing custom file search routines:

```toml
[project]
name = "blueprint_app"
version = "0.0.1"
cpp_version = "c++23"

# Explicit tracking maps to prevent redundant disk hits for shared packages
dependencies = [
    "../fmt"
]

[[targets]]
name = "blueprint_app"
type = "executable"
is_default = true
src = ["src/**/*.cpp"]
```

### Execution Lifecycle: Build to Runtime Launch

Mokai completely skips intermediate generation pipelines like Makefiles or Ninja configurations. It ingests source file structures directly, computes exact topological orderings, and issues low-level compiler toolchain dispatches in a single unified loop.

#### 1. Execute Parallel Graph Building
```bash
cd blueprint_app
mokai build
```

The underlying pipeline scans your workspace files, evaluates tracking registers for changed files, and distributes execution loops safely across every available hardware CPU worker thread.

#### 2. Triggering Immediate Runtime Executions
You don't need to manually hunt down output paths across separate profiles. Mokai parses target configurations directly, resolves fallback targets where `is_default = true`, verifies build signatures, and runs compiled binaries instantly:

```bash
mokai run
```

If you need to pass low-level arguments straight down into your executable's standard argument vectors (`argc`, `argv`), append them immediately following your targeted specification name:
```bash
mokai run blueprint_app --verbose --threads=8
```
